Justificaciones de Diseño - Entrega 4: Persistencia

En la cuarta entrega del desarrollo de MetaMapa, se realizó una transformación arquitectónica fundamental que migró 
el sistema de una arquitectura basada en memoria a una arquitectura persistente con capacidades avanzadas de análisis
 y normalización de datos. Esta evolución se justifica por la necesidad de garantizar la persistencia de información
  entre reinicios del sistema, permitir el manejo de grandes volúmenes de datos y soportar múltiples usuarios 
  simultáneos. Se implementó una arquitectura basada en Spring Data JPA con Hibernate como ORM y MySQL como motor 
  de base de datos, aplicando el patrón Repository para separar la lógica de acceso a datos de la lógica de negocio,
   facilitando la testabilidad y permitiendo cambios en la implementación de persistencia sin afectar el resto del 
   sistema.

La normalización de datos se implementó como una solución crítica al problema de inconsistencia en la representación
 de información proveniente de múltiples fuentes externas. Diferentes APIs y datasets utilizan convenciones distintas
  para categorías, provincias, fechas y coordenadas, lo que genera datos aparentemente diferentes que en realidad
   representan la misma información. Se desarrolló un servicio de normalización que aplica el patrón Strategy para 
   encapsular diferentes estrategias de estandarización, integrando automáticamente la normalización en el flujo de 
   creación de hechos para garantizar que todos los datos persistidos estén consistentes. Esta implementación sigue
    el principio de "Fail Fast", detectando y corrigiendo inconsistencias en el momento de la entrada al sistema.

El servicio de estadísticas se diseñó para responder preguntas específicas del negocio mediante la implementación
 del patrón Service Layer, que encapsula la lógica de análisis y proporciona respuestas a consultas como la provincia
  con mayor cantidad de hechos, la categoría más reportada, o la distribución temporal de eventos. Se aplicaron
   desnormalizaciones controladas del modelo relacional, incluyendo campos calculados y vistas materializadas a
    través de queries personalizadas con agregaciones, optimizando el rendimiento de consultas frecuentes.
     La estrategia de indexación se enfocó en campos de búsqueda y filtrado, implementando índices compuestos 
     para consultas complejas y manteniendo un equilibrio entre normalización para integridad referencial
      y desnormalización para optimización de análisis.

La funcionalidad de exportación de datos se implementó utilizando el patrón Strategy para manejar diferentes 
formatos, permitiendo la extensibilidad futura y separando la lógica de exportación de la capa de presentación.
 Se desarrolló un servicio de búsqueda por texto libre que utiliza el patrón Template Method para definir algoritmos
  de búsqueda en múltiples campos, mientras que el soporte multimedia se implementó mediante el patrón Composition
   para mantener la flexibilidad de hechos con o sin contenido asociado. La inyección de dependencias se utilizó
    extensivamente para desacoplar servicios, facilitar la testabilidad y permitir cambios de implementación sin 
    modificar el código existente.

La arquitectura resultante de la Entrega 4 establece una base sólida para futuras extensiones del sistema, 
incorporando persistencia robusta, normalización inteligente, análisis avanzado, búsqueda flexible y soporte multimedia. 
Los patrones de diseño aplicados proporcionan separación clara de responsabilidades, mantienen la testabilidad
 del código y establecen las bases para las entregas posteriores del trabajo práctico, cumpliendo con todos los 
 objetivos de incorporar nociones de persistencia relacional, mapeo objeto-relacional y desnormalizaciones del 
 modelo de datos.
